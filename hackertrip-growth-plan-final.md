# HackerTrip 增长方案 · 融合最终版

> 版本：Final（融合版）｜ 日期：2026-06-01
>
> **本版来源**：以「朋友版增长方案」的执行框架为主干（认领机制 / 赛后窗口 / 资源到人 / 里程碑验收），嵌入「PRD v2」的四个逻辑补丁（验证标识 / 回流率降级 / 隐私 opt-in / 同赛事榜单）。
>
> **标注约定**：`【沿用】` = 直接采纳朋友版；`【补丁】` = 来自 v2 的修正；`【删除】` = 移除有漏洞的设计。

**团队约束**：2 人。不做线下铺量、不做社区重运营、不做大规模 BD。一切依赖产品自身的传播力和赛事节奏的自然触发。

## 一、当前产品全貌

### 1.1 已有功能 【沿用】

| 功能 | 状态 | 一句话 |
|------|------|--------|
| 内容聚合（Explore） | 🟢 已上线 | AI 爬取全球黑客松，搜索+筛选+详情，数据量充足 |
| AI Skill 命令行 | 🟢 已上线 | curl 一行安装，终端内扫描项目匹配赛事，核心差异化 |
| Haki AI 对话 | 🟡 已上线待收窄 | 悬浮入口，定位从「发现+报名+推广」收窄为「赛事向导」 |
| 组织者入驻 | 🔴 空白 | 仅入口链接，无实际功能——等用户基数够了再启动 |

### 1.2 当前核心矛盾 【沿用】

> 功能都有了，但**用户不进来**。进来了也**没有留下的理由**。离开了也**没有回来的理由**。

| 问题 | 根因 |
|------|------|
| 为什么不进来 | 没有传播内容——命令行很好但没法分享，Explore 很好但没人知道 |
| 为什么不留下 | 没有「我的东西」——浏览完就走，平台上没有用户自己的数据资产 |
| 为什么不回来 | 没有回访钩子——没有关注、收藏、提醒，也没有赛后记录的需求承接 |

### 1.3 破局点 【沿用】

需要一个同时解决「进来、留下、回来」三个问题的功能 → **项目名片**。

## 二、裂变核心功能：项目名片

### 2.1 产品定位 【补丁｜定位校准】

> 项目名片是 HackerTrip 的**内容引擎**。让参赛者用 2 分钟录入项目信息，自动生成一张可传播的项目展示卡片。

> **【补丁】** 它带来的是**内容填充 + 品牌曝光 + 展示满足**（类比 LinkedIn 证书、Strava 成绩单），**不是病毒裂变引擎**。增长的真正动力来自「运营赛后代建 + 认领」（第三节），不是用户自发的好友裂变。原因见附录 A。

### 2.2 核心设计原则 【沿用 + 补丁】

| 原则 | 说明 |
|------|------|
| 赛后时刻触发 | 不要求用户「主动来」，而是在比赛刚结束、他本来就要晒项目的时刻，让 HackerTrip 成为最省力的选择 |
| 极低摩擦 | 必填字段 4 个，2 分钟内完成，手机端优先 |
| 一键分享 | 生成「图片 + 文案 + 链接」内容包，用户只需点击发送 |
| 先内容后注册 | 落地页永远是内容，不是注册墙——用户先看到价值，再决定行动 |
| 认领优于创建 | 冷启动阶段，主动帮获奖者建好名片，通知他们「认领」，比让他们「创建」门槛更低 |
| **可验证优先** | **【补丁】** 凡涉及「成绩 / 获奖 / 所属赛事」的字段，优先支持来源核验或标注验证状态，守住名片作为「背书」的价值 |
| **代建先授权** | **【补丁】** 运营代建的名片**默认草稿态、不公开索引**，本人认领后才正式公开（opt-in 软启动，避免隐私/姓名权风险） |

### 2.3 功能规格

#### 2.3.1 项目录入 【沿用】

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 项目名称 | 文本 ≤50 字 | ✅ | |
| 所属赛事 | 搜索联想 + 手动输入 | 选填 | 搜中即强关联；手填标记「未关联赛事」 |
| 参赛时间 | 月份选择器 | ✅ | 精确到月 |
| 技术栈标签 | 多选 + 自定义 | ✅ | AI/LLM、Web3、前端、后端、移动端、设计、硬件、数据、DevOps、其他 |
| 一句话介绍 | 文本域 ≤140 字 | ✅ | 「这个项目做了什么 + 最大亮点」 |
| 获奖情况 | 下拉 | 选填 | 冠军/亚军/季军/入围奖/参与 |
| 项目链接 | URL | 选填 | GitHub/Demo/官网 |

交互：单页表单 + 右侧实时预览，填写即看到卡片效果。

#### 2.3.2 名片图片（自动生成）【沿用 + 补丁】

规格：1200×630px PNG，适配 Twitter/即刻/朋友圈分享尺寸。

```
┌──────────────────────────────────────┐
│ 🏆 [获奖标识]        ✓ [验证标识]        │  ← 验证标识为补丁新增
│ [项目名称]  大号加粗 1-2 行             │
│ [一句话介绍] 常规字号最多 2 行           │
│ │ Python │ │ React │ │ AI │           │
│ 📍 [赛事名称] · [参赛时间]              │
│ ──────────────────────────────        │
│ HackerTrip │ hackertrip.space/@user    │
└──────────────────────────────────────┘
```

> **【补丁｜验证标识】** 卡片右上角显示验证状态，把「可信名片」和「自夸名片」在视觉上区分：
> - `✓ 已核实获奖`：运营代建 + 本人认领，或主办方确认 —— 绿色高信任
> - `✓ 链接可达`：项目链接经系统抓取确认存在 —— 中性
> - 无标识：纯自填未核验 —— 不显示徽章
>
> 这与「认领」机制天然契合：认领过的名片自动获得「✓ 已核实获奖」，是自发自填名片得不到的信任溢价，反过来激励用户走认领流程。

变体：
- **单项目卡片**：展示一个项目
- **个人战报**：≥3 个项目自动聚合为「N 个项目 · M 次获奖 · 跨越 X 年」+ 缩略图阵列
- **年度总结**：年末自动生成，全年参赛统计 + 技术栈分布

#### 2.3.3 一键分享内容包 【沿用】

用户创建名片后，系统生成一个分享内容包：

```
🖼️ 名片图片（1200×630 PNG）
📝 预写文案（可编辑）：
   「[项目名]」在 [赛事名] 拿了 [获奖] 🏆
   [一句话介绍]
   🔧 [标签1] · [标签2] · [标签3]
   👉 hackertrip.space/project/:id
🔗 分享渠道：[复制图片] [复制链接] [发 Twitter] [发即刻] [下载图片]
```

设计要点：用户只需点一个按钮，文案和图片都准备好了。分享行为是「发送」而非「创作」。

#### 2.3.4 项目详情页 【沿用 + 补丁】

URL：`hackertrip.space/project/:id`

| 区块 | 内容 |
|------|------|
| 头部 | 项目名称、一句话介绍、获奖标识、**验证标识**【补丁】 |
| 信息条 | 赛事名（可点击跳转赛事详情）、参赛时间、项目链接 |
| 技术标签 | 可点击搜索同技术栈项目 |
| 操作区 | 分享按钮、点赞 |
| 底部引导 | 「查看这场比赛的其他项目」→ Explore 赛事聚合页 + **同赛事项目榜单**【补丁】 |

#### 2.3.5 个人项目集 【沿用】

URL：`hackertrip.space/@username`

| 区块 | 内容 |
|------|------|
| 头部 | 头像、昵称、简介、技术标签 |
| 统计条 | N 个项目 · M 场比赛 · X 次获奖 |
| 项目网格 | 所有项目名片缩略图 |
| 空状态引导 | 「还没有项目？安装 AI Skill 找比赛 →」 |

#### 2.3.6 同赛事横向榜单 【补丁｜补自发增长钩子】

在项目详情页底部和赛事聚合页提供「同赛事项目榜单」（按点赞/浏览排序）。

> **为什么加这个**：朋友版的自发增长靠「产品力会让用户自己来」，但没给机制。横向榜单给了**同届选手一个精准的来建名片的理由**——「我的项目也在这场比赛，凭什么不上榜」。受众精准（同赛事 = 确定参赛者），是少数真正成立的自发增长钩子，用来支撑第 12 周「自发创建超过主动创建」的里程碑。

### 2.4 与现有功能的衔接 【沿用】

| 现有功能 | 项目名片如何衔接 |
|----------|------------------|
| Explore 活动详情页 | 底部展示「这场比赛诞生的项目」——赛事详情变成内容聚合页 |
| AI Skill 命令行 | 匹配结果底部引导「赛后为你的项目创建名片」——Skill 不只是一次性工具 |
| Haki AI 助手 | 对话中主动询问「你参加过哪些比赛？要不要记录下来？」 |

### 2.5 目标 【补丁｜指标重排】

| 指标 | 类别 | 上线后首月目标 |
|------|------|----------------|
| **真实名片总量**（认领 + 链接可达） | **北极星** 【补丁】 | ≥ 50 |
| 名片分享率（创建后分享的比例） | 核心 | ≥ 40% |
| 名片人均被访（详情页平均浏览） | 展示价值 【补丁】 | ≥ 20 |
| 新用户深度功能触达率（注册后用了搜索/Skill/Haki 任一） | 留存 | ≥ 25% |
| ~~分享回流率~~ | **观测项**（非 KPI）【补丁】 | 个位数，不设硬目标 |

> **【补丁｜为什么把回流率从 KPI 降为观测】** 回流依赖「好友→新参赛者」转化，而该人群结构性错配（附录 A），强设 KPI 会误导团队把精力投到失效路径。北极星换成「真实名片总量」，直接对应平台最缺的内容，且「真实/已验证」前缀杜绝刷量自夸。

## 三、裂变运营模式

### 3.1 核心逻辑 【沿用 + 补丁】

> 不是「拉人解锁功能」的工具型裂变，而是「展示我的作品」的身份型展示。
> 用户分享是因为他**本来就想晒项目**——HackerTrip 只是让晒得更省力、更好看。

> **【删除】** 移除朋友版 §3.2 飞轮里「好友看到 → 我也想要一张 → 注册」当作**增长主路径**的画法。保留「分享 = 品牌曝光」的定性，但不把好友裂变当作可指数放大的引擎。真正的增长引擎是下面的「赛后代建 + 认领」。

### 3.2 增长主引擎：赛后代建 + 认领 【沿用 — 朋友版的金点子】

```
比赛结束（天然触发点）
  ▼
运营用公开信息为 Top 5 获奖项目预建名片（草稿态）【补丁：草稿态】
  ▼
私信获奖者「恭喜！帮你做了张展示页，认领并编辑 →」
  ▼
认领 → 名片公开 + 获得「✓ 已核实获奖」徽章【补丁：验证徽章】
  ▼
获奖者大概率分享（卡片好看、文案现成、本来就想晒）
  ▼
品牌曝光 + 同赛事选手看到榜单 → 自发来建【补丁：榜单钩子】
```

**为什么这个方式有效**：
- 获奖者看到自己项目被漂亮展示，大概率会分享——「有人帮我的项目做了张卡片」
- 「认领」比「创建」门槛低——用户只需点开链接确认
- 每一条分享都来自真实获奖者，可信度极高

### 3.3 冷启动：3 步破冰（1-2 周）【沿用 + 补丁】

> 核心思路：不等人来，主动帮人做好名片，通知认领。

| 步骤 | 动作 | 说明 | 人力 |
|------|------|------|------|
| Step 1 | 盯一场近期结束的黑客松 | 选有影响力的比赛（AdventureX、腾讯云黑客松），关注结果公布时间 | 0.5 天 |
| Step 2 | 结果公布 24h 内，用公开信息为 Top 5 项目建名片**（草稿态）** | 从赛事官网/Twitter 获取项目名、介绍、技术栈、获奖、GitHub。底部标注「由 HackerTrip 整理」 | 1 天 |
| Step 3 | 私信获奖者认领 | 「恭喜获奖！我们在 HackerTrip 为你的项目建了展示页，可认领并编辑 →」。在 Twitter/即刻/微信联系。**认领后才公开**【补丁】 | 0.5 天 |

> **【补丁｜合规软启动】** 代建名片默认**草稿态、noindex、不进 Explore**，仅生成一个私享认领链接发给本人。本人认领后才正式公开。既保留「帮你整理好了」的惊喜，又规避未授权公开真人信息的姓名权/GDPR 风险。底线仍保留「一键删除」。

**预期效果**：一场比赛 5 个获奖项目，假设 3 人认领并分享，每人平均带来 50-100 次曝光 → 150-300 次品牌曝光 + 30-60 次点击进入。

### 3.4 持续增长：借赛事节奏（第 3 周起）【沿用】

| 阶段 | 动作 | 频率 | 说明 |
|------|------|------|------|
| 赛前 | 在 Explore 提前上架即将举办的活动 | 每周 | 让搜索用户看到「即将开始」的比赛 |
| 赛中 | 不打扰 | — | 参赛者在比赛，不需要干预 |
| 赛后（关键窗口） | 赛后 24-72h，主动帮获奖/亮眼项目建名片 + 通知认领 | 每场比赛 1 次 | 唯一需要人工介入的时刻 |
| 日常 | 社交媒体转发用户的名片分享 | 持续 | 官方账号转发 = 二次曝光 + 激励更多用户分享 |

节奏估算：每月盯 2-3 场重点黑客松，每场投入 1 天。其余时间做产品迭代。

### 3.5 传播内容矩阵 【沿用】

| 内容类型 | 触发时机 | 谁生产 | 传播渠道 |
|----------|----------|--------|----------|
| 单项目名片 | 每次用户录入 | 用户 + 系统 | Twitter/即刻/朋友圈 |
| 个人战报 | ≥3 个项目自动生成 | 系统 | 社交媒体 |
| 赛事项目聚合页 | 赛后项目名片积累 | 系统（自动聚合） | 搜索引擎/赛事社区 |
| 年度总结 | 年末 | 系统 | 全社交平台 |
| 「帮你整理好了」 | 赛后主动创建 | 运营 | 私信 + 社交媒体 |

### 3.6 运营度量 【补丁｜回流降级】

| 指标 | 定义 | 第 1 月 | 第 3 月 |
|------|------|---------|---------|
| 主动创建名片数 | 赛后运营帮用户建的数量 | ≥ 15 | ≥ 30 |
| 认领率 | 通知认领后实际认领的比例 | ≥ 50% | ≥ 50% |
| 用户自发创建名片数 | 用户自己注册并录入的数量 | ≥ 20 | ≥ 50 |
| 分享率 | 创建/认领后分享的比例 | ≥ 40% | ≥ 40% |
| 周活跃用户 | — | ≥ 100 | ≥ 300 |
| ~~分享回流注册数~~ | 观测项，不设硬目标【补丁】 | 观测 | 观测 |

## 六、风险与底线 【沿用 + 补丁】

| 风险 | 应对 |
|------|------|
| 代建名片被获奖者反感 | **【补丁】草稿态 + 私享认领链接（opt-in），认领后才公开**；标注「由 HackerTrip 整理」；私信语气是「恭喜+已整理」而非「要求使用」；提供一键删除 |
| 隐私/姓名权/GDPR | **【补丁】** 仅用公开信息；不公开索引直到认领；海外赛事尤其走 opt-in |
| 赛后窗口错过 | 提前关注赛事日程，设日历提醒；错过一场立刻切下一场 |
| 自填数据注水 | **【补丁】** 验证徽章体系（已核实获奖/链接可达）+ 链接可达性抓取 |

| 线性天花板（人力封顶） | **【补丁】** 同赛事榜单 + 个人战报/年度卡的自动化内容，把增长从「纯人工代建」逐步过渡到「自发 + 系统生成」 |

## 附录 A：为什么不把「好友裂变」当增长主路径

病毒系数 K = 人均分享触达 × 受众注册转化率，自循环增长要求 K > 1。

1. **人群错配**：获奖者社交圈是同事/朋友/同行，「自己也打黑客松」的比例极低 → 转化率结构性偏低。
2. **无双边激励**：分享者得展示满足，被分享者「什么都没得到」→ 炫耀式传播，非 referral 裂变。
3. **低频事件**：开发者一年参赛 1-4 次，名片是一次性录入，无持续回访理由。

**结论**：好友裂变贡献品牌曝光 + 少量精准回流，但 K < 1，不能当主引擎。增长主引擎是「赛后代建 + 认领」（人工冷启动）+「同赛事榜单」（精准自发），二者都精准命中「确定是参赛者」的人群。

## 七、一句话总结

> 两人团队不做铺量，只做一件事：**在每场黑客松结束后 24-72 小时内，用公开信息帮获奖者把项目名片做好（草稿态），私信通知认领。** 认领后名片公开、带「已核实」徽章、获奖者大概率分享。每一次分享是品牌曝光；同赛事选手看到榜单后自发来建，构成精准的自发增长。一个月盯 2-3 场，三个月验证模式，然后靠产品力（榜单 + 自动化内容）让用户自发创建超过人工代建。
