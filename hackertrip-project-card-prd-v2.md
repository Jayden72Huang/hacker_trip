## 一、这是一个怎样的产品

### 1.1 HackerTrip 是什么

HackerTrip 是一站式黑客松信息平台，核心能力包括：

| 能力 | 说明 |
|------|------|
| 信息聚合 | AI 自动爬取全国乃至全球黑客松活动，提供搜索、筛选、浏览 |
| 命令行集成 | 开发者通过一行命令安装 AI Skill，在终端中扫描项目、匹配赛事 |
| AI 对话助手 | Haki 赛事向导，对话式筛选和推荐黑客松 |
| 组织者工具 | 主办方入驻发布活动、数据看板（后期） |

### 1.2 项目名片是什么

项目名片是 HackerTrip 的**内容引擎与展示工具**。它不是独立产品，而是嵌入平台核心功能中的内容生产 + 传播模块。

> 一句话：参赛者花 2 分钟录入项目信息，系统自动生成一张可传播的项目名片，展示「这个项目是什么、参加了什么比赛、用了什么技术栈、取得了什么成绩」。

> **【v2 修订｜定位降级，诚实归位】** v1.0 把项目名片定位为「裂变钩子」，暗示它能驱动指数级用户增长。本版明确：**项目名片是「内容引擎 + 展示工具」，不是病毒裂变引擎**。它带来的是内容填充和品牌曝光（类比 LinkedIn 证书、Strava 成绩单），增长杠杆另由第六节的「B 端渠道分发」承担。原因见附录 A「为什么 C 端自发裂变在这里不成立」。

### 1.3 项目名片的形态

- **一张可分享的图片**：项目名 + 一句话介绍 + 技术标签 + 赛事名 + 获奖标识 + 验证标识 + HackerTrip 品牌标识 + 个人主页链接
- **一个公开项目详情页**：`hackertrip.space/project/:id`
- **一个个人项目集页面**：`hackertrip.space/@username`，聚合该用户所有项目名片

## 二、它对整个产品的价值

### 2.1 解决当前核心矛盾

> 四个核心功能已上线，但内容为空。Explore 活动数为 0，用户来了看不到任何东西，不会留下。

项目名片提供了**第一批有血肉的内容**，让平台不再是一个空壳。

> **【v2 修订｜内容来源换引擎】** v1.0 假设内容靠用户自发录入填满，但**冷启动期没有用户、没有内容是一个先有鸡还是先有蛋的死循环**。本版把第一批内容的来源明确为「赛事方批量邀请」（见 5.5）：与一场已结束/进行中的黑客松主办方合作，一次性导入几十名选手的真实项目，单场比赛即可填满一个赛事维度的 Explore。

### 2.2 三层价值定位

| 层级 | 对谁 | 价值 |
|------|------|------|
| 内容层 | 平台自身 | 填充 Explore 页面，让浏览体验完整；每个项目关联赛事，反哺活动数据 |
| 传播层 | 用户 | 自动生成可分享卡片，满足参赛者「展示作品」的天然需求，每次分享都是**品牌曝光**（非获客裂变） |
| 网络层 | 生态 | 项目名片聚合后形成「赛事-项目-参赛者」三元关联，为组织者端和企业端商业化提供数据基础 |

> **【v2 修订｜传播层与网络层的预期校准】**
> - 传播层：v1.0 称「每次分享都是裂变」，本版改为「每次分享是品牌曝光」。曝光会带来零星注册，但**不构成自循环增长**，不以此为北极星指标。
> - 网络层：v1.0 把「三元关联」当作自然形成的结果。本版强调它依赖**赛事归一化**——用户手填的「所属赛事」与平台爬取的赛事数据常对不上（名称不一致、赛事不在库、手填错漏）。三元关联是需要专门工程投入的能力，不是副产品（见 5.6）。

### 2.3 与核心功能的衔接

| 核心功能 | 项目名片如何增强 |
|----------|------------------|
| Explore 搜索筛选 | 活动详情页下展示「这场比赛诞生的项目」，浏览价值翻倍 |
| AI Skill 命令行 | 匹配赛事后引导「赛后为你的项目创建名片」 |
| Haki AI 助手 | 对话中推荐赛事时附带展示相关项目案例 |
| 组织者入驻 | 组织者可**主动邀请**参赛者建名片，并查看「谁参加了我的比赛、做了什么项目」（付费功能） |

## 三、产品设计思路

### 3.1 设计原则

| 原则 | 说明 |
|------|------|
| 极低摩擦 | 必填字段 ≤ 4 个，2 分钟内完成录入 |
| 独立闭环 | 录入 → 生成 → 展示 → 分享，一个用户从进来到完成分享，不需要依赖其他功能 |
| 自带传播 | 生成的内容天然适合社交分享，每张卡片底部带平台链接 |
| 渐进增强 | 1 个项目也能展示完整，3 个项目自动聚合统计数据，5 个项目解锁高级视图 |
| **可验证优先** | **【v2 新增】** 凡涉及「成绩 / 获奖 / 所属赛事」的字段，优先支持来源核验或标注验证状态，守住名片作为「背书」的价值 |

### 3.2 用户故事

> **核心故事（C 端）**：我是一个参加过黑客松的开发者。赛后我想展示自己的项目。我花 2 分钟填了几个字段，系统生成了一张很酷的项目名片。我把它分享到 Twitter，朋友看到觉得有意思——**我获得了展示满足感，平台获得了一次品牌曝光**。

> **【v2 新增｜B 端核心故事】**：我是一场黑客松的主办方。赛事结束后，我在 HackerTrip 后台一键群发邀请，让我的 80 名选手为各自的项目创建名片。选手因为「官方邀请 + 可获官方验证标识」而愿意填写，一周内平台多了 60 张**真实且经主办方背书**的项目名片，我的赛事详情页也变得丰满，方便我对外宣传赛事成果。

> **【v2 修订｜删除失效的裂变假设】** v1.0 用户故事的结尾是「朋友点链接进来，发现很多优秀项目，于是也注册创建了自己的」。本版删除这一跳——见附录 A，社交圈受众与「黑客松参赛者」严重错配，这一转化不可作为增长主路径。

### 3.3 功能模块

```
录入项目          生成名片          展示传播
  │                 │                 │
  ▼                 ▼                 ▼
7 字段表单  ───▶  卡片图片  ───▶  复制图片
2 分钟完成        详情页            复制链接
                  个人项目集        社交分享

         ▲
         │
   ┌─────┴──────┐
   │ 两个录入入口 │  【v2 新增】
   └────────────┘
   ① C 端自助录入（赛后自发）
   ② B 端批量邀请（主办方群发，主路径）
```

## 四、为什么一定要用这个产品

### 4.1 用户动机分析

黑客松参赛者有一个被忽视的刚需：「展示我的作品」。但目前：

| 现有方式 | 痛点 |
|----------|------|
| GitHub README | 只有代码，没有叙事，非技术人群看不懂 |
| 发 Twitter / 朋友圈 | 零散、不可检索、无法聚合 |
| Devpost / 活动官网 | 受限于单个赛事，无法跨赛事展示 |
| 个人博客 | 写作成本高，一篇长文动辄数小时 |

HackerTrip 项目名片解决的是：一个地方，集中展示你所有黑客松项目，2 分钟生成，一键分享。

### 4.2 不可替代性

| 竞品/替代方案 | 为什么不如项目名片 |
|---------------|--------------------|
| Devpost | 单赛事维度，无法跨赛事聚合个人项目集 |
| GitHub Profile | 代码导向，缺乏项目叙事和赛事关联 |
| LinkedIn 项目区 | 偏正式简历语境，不符合黑客松文化 |
| 个人博客/Notion | 需要主动维护，无模板无自动化 |

### 4.3 为什么现在必须做

1. **内容空窗期**：网站需要内容来承接用户，项目名片是最轻量的内容生产方式
2. **差异化信号**：在黑客松圈层建立「看优秀项目去哪？HackerTrip」的心智
3. **【v2 修订】渠道窗口期**：黑客松主办方普遍缺少「赛后选手作品归档与展示」的工具——这是切入 B 端、用 B 端流量启动 C 端的现成抓手，先发者建立合作壁垒

> **【v2 修订｜删除「传播破局」的伪命题】** v1.0 主张「项目名片是『推』的模式，用户分享出去，新用户被拉进来」。本版删除——分享的「推」无法精准命中目标用户（附录 A）。真正的「推」是 B 端渠道把名片精准推送给「确定是参赛者」的人群。

## 五、功能规格

### 5.1 项目录入

字段定义：

| 序号 | 字段名 | 类型 | 必填 | 说明 |
|------|--------|------|------|------|
| 1 | 项目名称 | 文本输入（≤50字） | ✅ | 自由输入 |
| 2 | 所属赛事 | 搜索联想 + 手动输入 | 选填 | 搜索已有活动，选中即建立强关联；选不中可手动输入（标记为「未关联赛事」） |
| 3 | 参赛时间 | 月份选择器 | ✅ | 精确到月 |
| 4 | 技术栈标签 | 多选标签 + 自定义 | ✅ | 预置：AI/LLM、Web3、前端、后端、移动端、设计、硬件、数据、DevOps、其他 |
| 5 | 一句话介绍 | 文本域（≤140字） | ✅ | 项目做了什么、最大亮点 |
| 6 | 获奖情况 | 下拉单选 | 选填 | 冠军/亚军/季军/入围奖/参与 |
| 7 | 项目链接 | URL 输入 | 选填 | GitHub / Demo / 官网 |

> **【v2 修订｜字段 2 关联逻辑显式化】** 「所属赛事」选中平台已有赛事时建立**强关联**（进入三元网络）；手动输入时标记为**「未关联赛事」**，仅作展示、不计入网络层统计，等待后续归一化（见 5.6）。这避免 v1.0 默认「手填即关联」造成的脏数据。

### 5.2 交互流程

```
Step 1: 输入项目名称 → 搜索关联赛事（可选）
Step 2: 选择参赛时间 → 选择技术栈标签
Step 3: 写一句话介绍 → 选获奖情况 → 填项目链接
Step 4: 预览名片 → 确认发布 → 弹出分享面板
```

设计要点：
- 单页表单，不分步（减少点击）
- 实时预览区：右侧展示名片生成效果，填写即预览
- 空状态引导：「还没有项目？去 Explore 找个比赛参加吧 →」

### 5.3 项目名片卡片（自动生成）

卡片规格：1200×630px（适配社交平台分享尺寸）

卡片布局：

```
┌──────────────────────────────────────┐
│ 🏆 [获奖标识]   ✓ [验证标识]            │  ← 验证标识为 v2 新增
│                                        │
│ [项目名称]                              │
│ 大号加粗字体，1-2 行                     │
│                                        │
│ [一句话介绍]                            │
│ 常规字号，最多 2 行，灰色                 │
│                                        │
│ │ AI │ │ React │ │ Web3 │  ← 技术标签   │
│                                        │
│ 📍 [赛事名称] · [参赛时间]               │
│ ──────────────────────────────────    │
│ HackerTrip │ hackertrip.space/@username │
└──────────────────────────────────────┘
```

> **【v2 新增｜验证标识】** 卡片右上角显示验证状态：
> - `✓ 主办方验证`（B 端邀请录入 / 主办方确认）—— 绿色高信任
> - `✓ 链接可达`（项目链接经系统抓取确认存在）—— 中性
> - 无标识（纯自填，未核验）—— 不显示徽章
> 目的：把「可信的名片」和「自夸的名片」在视觉上区分开，守住展示/背书价值（修正 v1.0 数据不可验证漏洞）。

卡片变体：
- **单项目卡片**：展示一个项目的完整信息
- **个人战报卡片**：聚合 ≥3 个项目时自动生成，展示「N 个项目 · M 次获奖 · 跨越 X 年」统计 + 项目缩略图阵列
- **年度总结卡片**：年末自动生成，展示当年参赛统计 + 技术栈分布

### 5.4 项目详情页

URL：`hackertrip.space/project/:id`

| 区块 | 内容 |
|------|------|
| 头部 | 项目名称、一句话介绍、获奖标识、验证标识 |
| 信息条 | 赛事名称、参赛时间、项目链接 |
| 技术标签 | 标签列表，可点击跳转搜索同技术栈项目 |
| 操作区 | 分享按钮、点赞按钮 |
| 关联推荐 | 「参加过同一赛事的其他项目」（≥3 项目时展示） + **【v2 新增】同赛事项目横向榜单入口** |

### 5.5 录入入口与赛事方批量邀请 【v2 新增核心】

> 这是修正 v1.0「冷启动靠手填 + 增长靠自发裂变」的核心改动。

**入口一：C 端自助录入**（赛后自发）—— 即 5.1~5.2 流程。

**入口二：B 端批量邀请**（主路径）：

| 能力 | 说明 |
|------|------|
| 主办方后台发起邀请 | 主办方上传/选择已结束赛事，生成专属邀请链接或导入选手名单 |
| 选手一键认领 | 选手通过邀请链接进入，赛事字段已预填（强关联），只需补项目名/介绍/链接 |
| 主办方验证背书 | 经此入口创建的名片自动带 `✓ 主办方验证` 标识 |
| 赛事成果页 | 主办方获得一个聚合页 `hackertrip.space/event/:id/projects`，可对外宣传赛事成果 |

**为什么这是杠杆**：主办方拥有「确定是参赛者」的完整名单 + 让选手展示成果的动机 + 官方触达渠道。一场比赛即可批量灌入数十张**真实、已验证、强关联**的名片，同时解决「冷启动内容」「数据可信」「赛事归一化」三个问题。

### 5.6 赛事归一化（网络层基础） 【v2 新增】

| 场景 | 处理 |
|------|------|
| 选中平台已有赛事 | 直接强关联 |
| 手填赛事名 | 进入归一化队列：模糊匹配平台赛事库（名称相似度 + 时间窗口），命中候选则提示用户确认归并；无命中保留为「未关联赛事」 |
| B 端邀请录入 | 天然强关联（赛事由主办方指定） |

> 修正 v1.0 把「三元关联」当自然结果的漏洞——它需要专门的匹配逻辑与数据治理。

### 5.7 同赛事横向榜单（双边动机） 【v2 新增】

在赛事成果页和项目详情页提供「同赛事项目榜单」（按点赞 / 浏览排序）。

> **为什么需要**：v1.0 的传播只有「分享者炫耀」单边动机。横向榜单给了**被分享者（尤其同届选手）一个也来建名片的理由**——「我的项目也在这场比赛，凭什么不上榜」。这是本功能里少数真正具备网络效应、且受众精准（同赛事=确定参赛者）的钩子。

### 5.8 分享能力

| 分享方式 | 说明 |
|----------|------|
| 复制图片 | 一键复制名片图片到剪贴板，直接粘贴到微信/飞书/Twitter |
| 复制链接 | 复制项目详情页链接 |
| 下载图片 | 保存 PNG 到本地 |
| 社交分享 | 预生成分享文案 + 图片，一键跳转 Twitter / 即刻 |

预生成分享文案模板：

```
「[项目名]」 — 我在 [赛事名] 做的项目 🚀
[一句话介绍]
🔧 [标签1] [标签2] [标签3]
👉 hackertrip.space/project/:id
```

## 六、增长模型 【v2 整体重写】

> v1.0 的「裂变飞轮」基于一个不成立的假设：参赛者把卡片分享到社交圈，圈内会涌现「也想要一个」的新参赛者。**附录 A 证明该假设的人群错配导致病毒系数 K < 1**。本版用「双引擎」替换单一飞轮。

### 6.1 增长双引擎

```
引擎一（主）：B 端渠道分发  —— 解决「从 0 到 1」
┌─────────────────────────────────────────────┐
│ 与黑客松主办方合作                              │
│   ▼                                           │
│ 主办方批量邀请选手建名片（精准触达确定参赛者）    │
│   ▼                                           │
│ 真实+已验证内容灌入 Explore + 赛事成果页         │
│   ▼                                           │
│ 内容池达到临界规模 → 平台具备「看项目」的留存价值  │
└─────────────────────────────────────────────┘

引擎二（辅）：C 端展示传播  —— 解决「品牌曝光」
┌─────────────────────────────────────────────┐
│ 选手分享名片到社交平台                          │
│   ▼                                           │
│ 品牌曝光（HackerTrip 心智渗透）                 │
│   ▼ （零星）                                   │
│ 其中真正的参赛者注册 → 自助建名片                │
│   ▼                                           │
│ 同赛事横向榜单 → 拉同届选手补建（精准网络效应）   │
└─────────────────────────────────────────────┘
```

> **关键认知修正**：增长的主动力是**引擎一（B 端把内容/用户精准灌进来）**，不是引擎二的自循环。引擎二贡献品牌曝光和少量精准回流，但**不被当作可指数放大的飞轮**。

### 6.2 运营节点

| 阶段 | 动作 | 目标 |
|------|------|------|
| 冷启动 | **【v2 修订】** 不再是「手动邀请 10-20 KOL」，而是**谈下 2-3 场已结束黑客松的主办方合作**，批量导入 100-200 个真实项目 | 内容池过临界 + 数据可信 |
| 赛事联动 | 在即将举办的黑客松，**与主办方约定赛后官方推送**「为你的项目创建名片」 | 借赛事流量 + 官方背书 |
| 周期性节点 | 年末生成「年度战报」、大赛后生成「赛后总结卡」 | 周期性回流 |
| 同赛事挑战 | **【v2 修订】** 从「和朋友比战绩」改为「**同赛事选手上榜竞争**」（受众精准） | 精准网络效应 |
| 社区联动 | 在技术社区（掘金/思否/V2EX）发起「晒出你的黑客松项目」话题 | 外部引流 |

> **【v2 修订｜冷启动彻底换法】** v1.0 的「手动邀请 KOL」既慢又解决不了内容规模与可信度。本版冷启动 = **B 端主办方合作**，一次合作 = 一整场比赛的真实项目，是数量级的差异。

### 6.3 风险与依赖 【v2 新增】

| 风险 | 缓解 |
|------|------|
| 谈不下主办方合作 | 先做 1 场样板（哪怕免费/自营小赛事），用成果页 demo 撬动后续合作 |
| 选手不响应主办方邀请 | 邀请文案强调「官方验证标识 + 永久作品集」；主办方在颁奖/结营时统一引导 |
| C 端自助录入量低 | 这是**预期内**的——C 端不是主引擎，不为低自助量焦虑 |
| 自填数据注水 | 验证标识体系（5.3）+ 链接可达性抓取 |

## 七、度量指标 【v2 重写】

> v1.0 以「分享回流率 ≥15%」为核心指标，建立在裂变假设上（附录 A 证明该转化率不现实）。本版改用**内容覆盖**与**展示价值**为北极星，把传播指标降级为参考。

### 7.1 北极星与核心指标（首月）

| 类别 | 指标 | 定义 | 目标（首月） |
|------|------|------|--------------|
| **北极星** | 真实项目名片数 | 经验证（B 端/链接可达）的名片总数 | **150+** |
| 内容覆盖 | 有项目的赛事数 | Explore 中至少关联 1 个项目的赛事占比 | 头部赛事覆盖 ≥ 10 场 |
| 渠道 | 主办方合作数 | 已落地批量邀请的赛事方数量 | **≥ 2** |
| 展示价值 | 名片人均被访 | 单张名片详情页平均浏览量 | ≥ 20 |
| 留存信号 | 个人项目集创建率 | 注册用户中创建 ≥1 张名片的比例 | ≥ 30% |

### 7.2 参考指标（不作 KPI，仅观测）

| 指标 | 定义 | 说明 |
|------|------|------|
| 分享率 | 创建后执行分享动作的比例 | 观测传播意愿，不强求 |
| 分享回流率 | 通过分享链接进入并注册的占比 | **【v2 修订】** 从 KPI 降为观测项，预期个位数，不为其设硬目标 |

> **【v2 修订｜为什么换北极星】** 「真实项目名片数」直接对应平台最缺的「内容」，且「真实/已验证」前缀杜绝刷量自夸。「主办方合作数」是主引擎的领先指标。回流率因人群错配不可控，强行设为 KPI 会误导团队把资源投到失效的自发裂变上。

## 附录 A：为什么 C 端自发裂变在这里不成立

**病毒系数 K = 人均分享触达数 × 受众注册转化率**，自循环增长要求 K > 1。

1. **人群错配**：参赛者分享到社交圈，受众是同事/朋友/同行，其中「自己也参加过黑客松、也有项目想展示」的比例极低 → 转化率被结构性压低。
2. **无双边激励**：分享者得到展示满足，被分享者「什么都没得到」，缺乏注册动机 → 这是炫耀式传播，不是 referral 裂变。
3. **低频事件**：开发者一年参赛 1-4 次，名片是一次性录入行为，无持续回访理由 → 难以靠用户密度撑起飞轮。
4. **冷启动空池**：飞轮靠「看到更多优秀项目」吸引，但启动期内容池近乎空 → 钩子失效。

**结论**：上述四点叠加使 K 远小于 1。C 端传播应被定位为**品牌曝光 + 少量精准回流**，而非增长主引擎。增长主引擎是 B 端渠道分发（第六节）。

## 一句话总结

> **【v2】** 项目名片是 HackerTrip 的**内容引擎**：用 **B 端主办方批量邀请**精准灌入真实、已验证的项目，让 Explore 不再是空表、让组织者有数据可看；C 端的低摩擦生成与分享提供品牌曝光与展示满足。它修正了 v1.0「靠 C 端自发裂变增长」的核心误判，把增长杠杆放在真正握有参赛者人群的主办方手里。
