HackerTrip 增长方案 · 融合最终版

团队约束：2 人。不做线下铺量、不做社区重运营、不做大规模 BD。一切依赖产品自身的传播力和赛事节奏的自然触发。

一、当前产品全貌

1.1 已有功能 【沿用】

功能状态一句话
内容聚合（Explore）🟢 已上线AI 爬取全球黑客松，搜索+筛选+详情，数据量充足
AI Skill 命令行🟢 已上线curl 一行安装，终端内扫描项目匹配赛事，核心差异化
Haki AI 对话🟡 待收窄悬浮入口，定位从「发现+报名+推广」收窄为「赛事向导」
组织者入驻🔴 空白仅入口链接，无实际功能——等用户基数够了再启动

1.2 当前核心矛盾 【沿用】

功能都有了，但用户不进来。进来了也没有留下的理由。离开了也没有回来的理由。
问题根因
为什么不进来没有传播内容——命令行很好但没法分享，Explore 很好但没人知道
为什么不留下没有「我的东西」——浏览完就走，平台上没有用户自己的数据资产
为什么不回来没有回访钩子——没有关注、收藏、提醒，也没有赛后记录的需求承接

1.3 破局点 【沿用】

需要一个同时解决「进来、留下、回来」三个问题的功能 → 项目名片。

二、裂变核心功能：项目名片

2.1 产品定位 【补丁｜定位校准】

项目名片是 HackerTrip 的内容引擎。让参赛者用 2 分钟录入项目信息，自动生成一张可传播的项目展示卡片。
【补丁】 它带来的是内容填充 + 品牌曝光 + 展示满足（类比 LinkedIn 证书、Strava 成绩单），不是病毒裂变引擎。增长的真正动力来自「运营赛后代建 + 认领」（第三节），不是用户自发的好友裂变。原因见附录 A。

2.2 核心设计原则 【沿用 + 补丁】

原则说明
赛后时刻触发不要求用户「主动来」，而是在比赛刚结束、他本来就要晒项目的时刻，让 HackerTrip 成为最省力的选择
极低摩擦必填字段 4 个，2 分钟内完成，手机端优先
一键分享生成「图片 + 文案 + 链接」内容包，用户只需点击发送
先内容后注册落地页永远是内容，不是注册墙——用户先看到价值，再决定行动
认领优于创建冷启动阶段，主动帮获奖者建好名片，通知他们「认领」，比让他们「创建」门槛更低
可验证优先【补丁】 凡涉及「成绩 / 获奖 / 所属赛事」的字段，优先支持来源核验或标注验证状态，守住名片作为「背书」的价值
代建先授权【补丁】 运营代建的名片默认草稿态、不公开索引，本人认领后才正式公开（opt-in 软启动，避免隐私/姓名权风险）

2.3 功能规格

2.3.1 项目录入 【沿用】

字段类型必填说明
项目名称文本 ≤50 字✅
所属赛事搜索联想 + 手动选填搜中即强关联；手填标记「未关联赛事」
参赛时间月份选择器✅精确到月
技术栈标签多选 + 自定义✅AI/LLM、Web3、前端、后端、移动端、设计、硬件、数据、DevOps、其他
一句话介绍文本域 ≤140 字✅「这个项目做了什么 + 最大亮点」
获奖情况下拉选填冠军/亚军/季军/入围奖/参与
项目链接URL选填GitHub/Demo/官网

交互：单页表单 + 右侧实时预览，填写即看到卡片效果。

2.3.2 名片图片（自动生成）【沿用 + 补丁】

规格：1200×630px PNG，适配 Twitter/即刻/朋友圈分享尺寸。

┌──────────────────────────────────────┐
│ 🏆 [获奖标识]        ✓ [验证标识]       │  ← 验证标识为补丁新增
│ [项目名称]  大号加粗 1-2 行            │
│ [一句话介绍] 常规字号最多 2 行          │
│ │ Python │ │ React │ │ AI │           │
│ 📍 [赛事名称] · [参赛时间]             │
│ ──────────────────────────────        │
│ HackerTrip │ hackertrip.space/@user    │
└──────────────────────────────────────┘
【补丁｜验证标识】 卡片右上角显示验证状态，把「可信名片」和「自夸名片」在视觉上区分：
  • ✓ 已核实获奖：运营代建 + 本人认领，或主办方确认 —— 绿色高信任
  • ✓ 链接可达：项目链接经系统抓取确认存在 —— 中性
  • 无标识：纯自填未核验 —— 不显示徽章
这与「认领」机制天然契合：认领过的名片自动获得「✓ 已核实获奖」，是自发自填名片得不到的信任溢价，反过来激励用户走认领流程。

变体：单项目卡片 / 个人战报（≥3 个项目聚合）/ 年度总结（年末自动生成）。

2.3.3 一键分享内容包 【沿用】

🖼️ 名片图片（1200×630 PNG）
📝 预写文案（可编辑）：
   「[项目名]」在 [赛事名] 拿了 [获奖] 🏆
   [一句话介绍]
   🔧 [标签1] · [标签2] · [标签3]
   👉 hackertrip.space/project/:id
🔗 分享渠道：[复制图片] [复制链接] [发 Twitter] [发即刻] [下载图片]

设计要点：用户只需点一个按钮，文案和图片都准备好了。分享行为是「发送」而非「创作」。

2.3.4 项目详情页 【沿用 + 补丁】

区块内容
头部项目名称、一句话介绍、获奖标识、验证标识【补丁】

2.3.5 个人项目集 【沿用】

URL：hackertrip.space/@username。头部（头像/昵称/简介/标签）+ 统计条（N 项目 · M 比赛 · X 获奖）+ 项目网格 + 空状态引导。

2.3.6 同赛事横向榜单 【补丁｜补自发增长钩子】

在项目详情页底部和赛事聚合页提供「同赛事项目榜单」（按点赞/浏览排序）。

为什么加这个：朋友版的自发增长靠「产品力会让用户自己来」，但没给机制。横向榜单给了同届选手一个精准的来建名片的理由——「我的项目也在这场比赛，凭什么不上榜」。受众精准（同赛事 = 确定参赛者），是少数真正成立的自发增长钩子，用来支撑第 12 周「自发创建超过主动创建」的里程碑。

三、裂变运营模式

3.1 核心逻辑 【沿用 + 补丁】

不是「拉人解锁功能」的工具型裂变，而是「展示我的作品」的身份型展示。用户分享是因为他本来就想晒项目——HackerTrip 只是让晒得更省力、更好看。
【删除】 移除朋友版飞轮里「好友看到 → 我也想要一张 → 注册」当作增长主路径的画法。保留「分享 = 品牌曝光」的定性，但不把好友裂变当作可指数放大的引擎。真正的增长引擎是下面的「赛后代建 + 认领」。

3.2 增长主引擎：赛后代建 + 认领 【沿用 — 朋友版的金点子】

比赛结束（天然触发点）
  ▼
运营用公开信息为 Top 5 获奖项目预建名片（草稿态）【补丁】
  ▼
私信获奖者「恭喜！帮你做了张展示页，认领并编辑 →」
  ▼
认领 → 名片公开 + 获得「✓ 已核实获奖」徽章【补丁】
  ▼
获奖者大概率分享（卡片好看、文案现成、本来就想晒）
  ▼
品牌曝光 + 同赛事选手看到榜单 → 自发来建【补丁】

为什么有效：① 获奖者看到项目被漂亮展示，大概率分享；② 「认领」比「创建」门槛低，只需点开链接确认；③ 每条分享来自真实获奖者，可信度极高。

3.3 冷启动：3 步破冰（1-2 周）【沿用 + 补丁】

步骤动作说明人力
Step 1盯一场近期结束的黑客松选有影响力的比赛（AdventureX、腾讯云黑客松），关注结果公布时间0.5 天
Step 2结果公布 24h 内，为 Top 5 项目建名片（草稿态）从赛事官网/Twitter 获取项目名、介绍、技术栈、获奖、GitHub。底部标注「由 HackerTrip 整理」1 天
Step 3私信获奖者认领「恭喜获奖！我们为你的项目建了展示页，可认领并编辑 →」。认领后才公开【补丁】0.5 天
【补丁｜合规软启动】 代建名片默认草稿态、noindex、不进 Explore，仅生成私享认领链接发给本人。本人认领后才正式公开。既保留「帮你整理好了」的惊喜，又规避未授权公开真人信息的姓名权/GDPR 风险。底线仍保留「一键删除」。

预期效果：一场比赛 5 个获奖项目，假设 3 人认领并分享，每人带来 50-100 次曝光 → 150-300 次品牌曝光 + 30-60 次点击进入。

3.4 持续增长：借赛事节奏（第 3 周起）【沿用】

阶段动作频率
赛前在 Explore 提前上架即将举办的活动每周
赛中不打扰—
赛后（关键窗口）赛后 24-72h，主动帮获奖项目建名片 + 通知认领每场 1 次
日常官方账号转发用户的名片分享持续

节奏估算：每月盯 2-3 场重点黑客松，每场投入 1 天。其余时间做产品迭代。

六、风险与底线 【沿用 + 补丁】

风险应对
代建名片被获奖者反感【补丁】草稿态 + 私享认领链接（opt-in），认领后才公开；标注「由 HackerTrip 整理」；私信语气是「恭喜+已整理」而非「要求使用」；提供一键删除
隐私/姓名权/GDPR【补丁】 仅用公开信息；不公开索引直到认领；海外赛事尤其走 opt-in
赛后窗口错过提前关注赛事日程，设日历提醒；错过一场立刻切下一场
自填数据注水【补丁】 验证徽章体系（已核实获奖/链接可达）+ 链接可达性抓取
分享率低于预期迭代名片模板，做 A/B 测试；检查分享内容包文案
线性天花板（人力封顶）【补丁】 同赛事榜单 + 个人战报/年度卡自动化内容，把增长从「纯人工代建」过渡到「自发 + 系统生成」
两人忙不过来赛后运营每月只盯 2-3 场；日常靠自动化（系统生成内容、官方号转发）

附录 A：为什么不把「好友裂变」当增长主路径 【补丁新增】

病毒系数 K = 人均分享触达 × 受众注册转化率，自循环增长要求 K > 1。
  1. 人群错配：获奖者社交圈是同事/朋友/同行，「自己也打黑客松」的比例极低 → 转化率结构性偏低。
  2. 无双边激励：分享者得展示满足，被分享者「什么都没得到」→ 炫耀式传播，非 referral 裂变。
  3. 低频事件：开发者一年参赛 1-4 次，名片是一次性录入，无持续回访理由。
结论：好友裂变贡献品牌曝光 + 少量精准回流，但 K < 1，不能当主引擎。增长主引擎是「赛后代建 + 认领」+「同赛事榜单」，二者都精准命中「确定是参赛者」的人群。

七、一句话总结

两人团队不做铺量，只做一件事：在每场黑客松结束后 24-72 小时内，用公开信息帮获奖者把项目名片做好（草稿态），私信通知认领。 认领后名片公开、带「已核实」徽章、获奖者大概率分享。每一次分享是品牌曝光；同赛事选手看到榜单后自发来建，构成精准的自发增长。一个月盯 2-3 场，三个月验证模式，然后靠产品力让用户自发创建超过人工代建。
